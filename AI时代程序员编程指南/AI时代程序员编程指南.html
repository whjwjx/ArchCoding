Chapter 01 — AI时代编程新范式

本章将帮助你理解AI如何改变编程方式，建立正确的人机协作思维，明确程序员在AI时代的新角色定位。

AI辅助编程
AI-Assisted Programming / 使用AI工具辅助软件开发过程

AI辅助编程是指利用大语言模型（如GPT-4、Claude、GitHub Copilot等）来辅助程序员完成编码、调试、文档编写等工作的编程方式。这不是让AI替代程序员，而是让AI成为程序员的"超级助手"。

类比：就像建筑师有了智能设计助手，可以快速生成建筑草图、计算结构承重、优化空间布局，但最终的创意、决策和责任仍然在建筑师手中。
提示：AI辅助编程的核心是"人机协作"，不是"AI替代"。程序员需要从"代码编写者"转变为"代码指挥者"。
人机协作
Human-AI Collaboration / 人类与AI协同完成复杂任务的工作模式

人机协作是一种新型工作模式，人类负责顶层决策、架构方向、规范标准、风险边界、业务逻辑，AI负责细节落地、代码编写、文档产出、重复劳动、问题排查。主次一旦颠倒，AI产出必然混乱、不规范、漏洞多、无法维护；主次摆正，就能做到高效、稳定、可迭代、工程化。

类比：就像导演与剧组的关系。导演（人）负责创意、剧本、选角、整体把控，剧组（AI）负责摄影、灯光、道具、后期制作。导演不需要亲自扛摄像机，但必须清楚自己想要什么画面。
提示：人机协作的关键是"人抓大脑，AI做手脚"。人掌握核心控制权，AI负责执行落地。
角色定位
Role Positioning / 在AI时代程序员需要重新定义自己的职责和价值

在AI时代，程序员的角色从"代码编写者"转变为"产品经理+架构师+UI总监+项目总指挥"。程序员只做"定方向、定标准、定契约、拍板确认"，剩下所有落地干活全丢给AI。具体来说，程序员的核心工作包括：

  1. 敲定业务需求：定做什么、不做什么，有哪些页面、哪些功能、角色权限
  2. 敲定整体技术架构：选前后端栈、项目分层、部署方式、整体框架结构
  3. 敲定UI全局风格规范：主色调、组件库、排版风格、圆角/间距/字体规范、整体布局
  4. 敲定数据库+接口契约：表结构、字段、关联关系；每个模块的接口路径、入参出参
  5. 定代码规范+项目目录骨架：全局目录、命名规则、返回体格式、代码分层
类比：就像从"建筑工人"升级为"建筑设计师+项目经理"。以前你需要亲自搬砖、砌墙，现在你只需要画图纸、定标准、监督施工质量。
提示：角色转变的核心是从"执行者"变为"决策者"。你不需要深陷写业务代码、画UI、抠样式、写接口这种琐事，只把控顶层和规范就行。

Chapter 02 — 人机协作五大铁律

本章将介绍人机协作的五条核心铁律，这些铁律适用于所有开发场景，是避免翻车、保证质量的基础保障。

需求拆解原则
Requirement Decomposition Principle / 需求不拆细，绝不让AI动手

模糊指令（做一个后台、写一个后端）= AI产出垃圾；必须拆解到模块→功能→接口→字段最小粒度。这是人机协作的第一条铁律，也是最容易被忽视的原则。

类比：就像装修房子，你不能只说"给我装修一下"，而要说清楚"客厅铺木地板，卧室刷浅蓝色墙漆，厨房装整体橱柜，卫生间做干湿分离"。指令越具体，结果越符合预期。
提示：需求拆解的标准是"AI不需要猜测，只需要执行"。如果AI还需要问你"具体要什么功能"，说明需求还不够细。
规范先行原则
Standards First Principle / 规范先行，开发后置

先把目录、命名、接口、返回体、代码风格、安全规则全部定好，发给AI强制执行，不允许AI自创风格。规范是项目的"宪法"，所有代码都必须遵守。

类比：就像开车前先定交通规则。红灯停、绿灯行、靠右行驶、限速60，这些规则必须先定好，否则路上必然混乱。代码规范就是项目的"交通规则"。
提示：规范一旦定好，就要严格执行。如果AI产出的代码不符合规范，必须打回重写，不能妥协。
任务拆分原则
Task Decomposition Principle / 一次只给AI一个小任务

禁止让AI一次性写完整项目、整套架构；按"模块→接口→函数"分批交付，可控、易纠错。这是保证代码质量的关键原则。

类比：就像吃饭要一口一口吃，不能一口吃成胖子。给AI的任务太大，它就会"消化不良"，产出的代码质量必然下降。
提示：任务拆分的粒度标准是"一次对话能完成，一次测试能验证"。如果一个任务需要多轮对话才能说清楚，说明任务还不够小。
架构控制原则
Architecture Control Principle / 架构&技术栈人全权决定，AI无权选型

用什么框架、数据库、中间件、部署方案人定，AI只负责基于选定技术栈落地实现。这是保证项目可控性的核心原则。

类比：就像建房子，建筑师决定用钢筋混凝土结构还是木结构，工人只需要按照图纸施工。如果让工人自己决定用什么材料，房子可能建歪。
提示：技术选型是战略决策，影响项目的长期发展。AI不了解项目的业务背景、团队技术栈、运维能力，不能让它做这种决策。
人工审核原则
Manual Review Principle / AI所有产出，人工必审核兜底

业务逻辑、权限控制、安全漏洞、边界异常，AI考虑不全，必须人工校验再合并使用。这是质量保障的最后一道防线。

类比：就像医生开处方，AI可以帮忙写病历、查资料，但最终的处方必须由医生审核签字。因为医生了解患者的具体情况，AI不了解。
提示：人工审核的重点是业务逻辑、权限控制、安全漏洞、边界异常。这些是AI最容易出错的地方，也是最容易造成严重后果的地方。

Chapter 03 — 需求梳理阶段

本章将介绍在需求梳理阶段如何让人机协作实现"全面无遗漏"，确保需求分析的质量和完整性。

PRD文档
Product Requirements Document / 产品需求文档，描述产品功能、用户角色、业务流程的文档

PRD文档是需求梳理阶段的核心产出物，包含项目背景、用户角色、核心功能、非功能需求、异常场景、版本迭代规划。AI可以将你的口语需求，转成标准PRD文档、用户故事、功能清单。

类比：就像建筑图纸，PRD文档是项目的"蓝图"，所有人（包括AI）都按照这个蓝图施工。
提示：PRD文档的关键是"结构清晰，不要随意增删业务范围"。AI容易自作主张添加或删除功能，需要人工把控。
用户故事
User Story / 从用户角度描述需求的简短描述

用户故事是敏捷开发中常用的需求描述方式，格式为"作为一个[角色]，我想要[功能]，以便[目的]"。AI可以帮助梳理正向+异常边界场景，AI擅长补人类容易忽略的边缘情况。

类比：就像编剧写剧本，每个场景都要考虑"角色想做什么，为什么这么做，会遇到什么问题"。
提示：AI梳理的用户故事需要人工审核，特别是业务逻辑和边界条件，AI可能理解有偏差。
功能清单
Feature List / 列出所有功能点的清单

功能清单是需求梳理的重要产出，包含所有功能点、优先级、依赖关系。AI可以帮助输出流程图、业务结构图、功能脑图，拆解迭代版本：一期/二期功能拆分。

类比：就像购物清单，列清楚要买什么、先买什么、后买什么，避免遗漏。
提示：功能清单要按照"核心功能→辅助功能→优化功能"的优先级排序，AI可能把所有功能都当成同等重要。
指挥话术模板
Command Template / 标准化的AI指令模板

需求梳理阶段的标准Prompt模板：

基于我提供的业务需求，按照互联网标准PRD格式输出，包含：
项目背景、用户角色、核心功能、非功能需求、异常场景、版本迭代规划，
结构清晰，不要随意增删业务范围。
类比：就像给AI一个"填空题"，让它按照固定格式填写内容，避免AI自由发挥。
提示：使用模板时，要把你的业务需求具体描述清楚，AI才能产出高质量的PRD文档。

Chapter 04 — 架构与技术设计

本章将介绍在架构与技术设计阶段如何实现"结构规范、架构稳定"，确保项目的技术基础扎实可靠。

技术栈选型
Technology Stack Selection / 选择项目使用的技术组合

前后端技术栈（如：React+FastAPI+MySQL+Redis）、项目架构模式（单体/分层架构/微服务）、整体模块拆分（用户、权限、文件、业务、日志等）、部署环境（Linux服务器、Docker、反向代理、域名HTTPS）、安全方案（鉴权JWT、接口防刷、跨域、数据脱敏）等，这些都必须由人来决定。

类比：就像选择建筑材料，用钢筋混凝土还是木结构，用砖墙还是玻璃幕墙，这些决策影响建筑的承重、成本、工期、维护难度。
提示：技术选型要考虑团队能力、项目规模、运维成本、长期维护等因素，AI不了解这些背景，不能让它做决策。
模块拆分
Module Decomposition / 将系统拆分为独立的业务模块

AI可以输出整体项目目录结构（强制统一，后续所有代码都按此目录）、数据库设计（ER图、数据表、字段设计、索引、关联关系）、全局基础架构封装（全局异常处理、日志封装、配置中心、跨域配置）、绘制架构拓扑图、数据流图、权限结构图。

类比：就像把大房子分成客厅、卧室、厨房、卫生间，每个空间有独立的功能，但又相互连接。
提示：目录结构一旦确定，全程不许AI私自新增/修改文件夹。这是保证项目结构统一的关键。
数据库设计
Database Design / 设计数据表结构、字段、索引、关联关系

数据库设计是架构阶段的核心工作，包括ER图、数据表、字段设计、索引、关联关系。AI可以帮助设计数据库，但最终的表结构必须由人来审核定稿。

类比：就像设计建筑的承重结构，哪里是柱子、哪里是梁、哪里是墙，这些决定了建筑的稳定性和扩展性。
提示：数据库设计要考虑业务增长、查询性能、数据一致性等因素。AI设计的表结构可能存在范式问题、索引缺失、关联关系不合理等问题，需要人工审核。
全局架构封装
Global Architecture Encapsulation / 封装全局通用的基础功能

全局架构封装包括全局异常处理、日志封装、配置中心、跨域配置等。这些是项目的基础设施，所有模块都会使用。

类比：就像建筑的供水供电系统，每个房间都需要用水用电，所以需要统一设计和施工。
提示：全局架构封装要考虑扩展性、性能、安全性。AI可能只考虑当前需求，忽略未来的扩展。

Chapter 05 — 接口设计规范

本章将介绍如何设计规范的接口，实现前后端彻底对齐、统一标准，从源头保证项目的规范性。

RESTful规范
RESTful API / 符合REST架构风格的接口设计规范

RESTful是一种接口设计风格，强调资源导向、统一接口、无状态。人负责定接口风格：RESTful规范；AI负责按模块批量生成OpenAPI/Swagger接口文档。

类比：就像交通规则，红灯停、绿灯行、靠右行驶，所有人遵守同一套规则，交通才能顺畅。
提示：RESTful规范的核心是"资源导向"，URL表示资源，HTTP方法表示操作。AI可能不完全理解业务，需要人工把控接口设计。
统一返回体
Unified Response Format / 所有接口使用统一的返回格式

全局统一返回体格式、全局状态码定义是接口规范的核心。所有接口响应格式统一、报错格式统一，项目从源头就规范，不会后期乱糟糟。

类比：就像快递包裹，都有统一的格式：收件人、寄件人、物品、重量、费用，这样快递员才能高效处理。
提示：统一返回体要包含：状态码、消息、数据三个部分。状态码要分类：成功、客户端错误、服务端错误。
接口契约
API Contract / 接口的约定，包括路径、请求方式、入参、出参、校验规则

接口契约是前后端的"合同"，定义了接口的所有细节。人负责定鉴权方式、请求头公共参数、分页通用规则；AI负责逐个接口定义：路径、请求方式、入参、出参、校验规则。

类比：就像合同，双方签字前要把所有条款都谈清楚，签字后就不能随意更改。
提示：接口契约要先定好，再写代码。这样可以避免前后端联调时的扯皮和返工。
Swagger文档
Swagger / OpenAPI规范的实现，用于描述和文档化RESTful API

AI可以按模块批量生成OpenAPI/Swagger接口文档，生成前后端对接接口清单，避免联调扯皮。

类比：就像产品说明书，详细描述了每个接口的功能、参数、返回值，前后端都按照这个说明书开发。
提示：Swagger文档要实时更新，接口变更时同步更新文档。AI生成的文档可能不完整，需要人工补充。

Chapter 06 — 编码开发实战

本章将介绍日常编码开发中如何实现"高效写代码、风格统一稳定"，这是人机协作最频繁的环节。

强制约束文件
Mandatory Constraint Files / 提前定义的规范文件，AI必须遵守

提前一次性发给AI这些规范，后续所有编码遵守：项目完整目录结构、命名规范（变量/函数/表名/接口名）、代码注释规范、分层规范（Controller/Service/Dao/Model）、代码格式规范（PEP8、Eslint、Prettier等）。

类比：就像员工手册，新员工入职前先学习公司的规章制度，工作时就按照手册执行。
提示：强制约束文件要一次性给AI，不要分多次给，否则AI容易遗漏或混淆。
任务拆分策略
Task Decomposition Strategy / 将开发任务拆分为小任务的策略

正确顺序：全局公共类→基础工具类→数据模型→单个接口业务逻辑→前端页面/组件。一次只让AI写一个接口/一个文件。

类比：就像搭积木，先搭地基，再搭柱子，然后搭墙，最后搭屋顶。顺序错了，房子就搭不起来。
提示：任务拆分要考虑依赖关系，先做被依赖的模块，后做依赖其他模块的部分。
代码规范
Code Standards / 代码编写必须遵守的规范

代码规范包括命名规范、注释规范、格式规范、分层规范等。AI产出的代码必须符合这些规范，否则打回重写。

类比：就像写作文要遵守语法规则，句子要通顺，段落要清晰，否则读者看不懂。
提示：代码规范要具体可执行，不要用模糊的描述。比如"变量名要有意义"不如"变量名要能表达变量的用途，不使用单字母变量名（循环变量除外）"。
AI产出要求
AI Output Requirements / 对AI产出的代码质量要求

AI产出的代码必须满足：代码带完整注释、可直接运行、依赖明确、逻辑分层清晰、解耦、不写冗余代码、不私自引入新依赖。

类比：就像工厂的产品，必须符合质量标准才能出厂，否则就是次品。
提示：AI产出的代码要人工审核，特别是业务逻辑、权限控制、安全漏洞、边界异常。

Chapter 07 — 单接口vs批量开发

本章将深入讨论"一次只让AI写一个接口"的效率问题，分析单接口开发和批量开发的优劣，给出最优的混合工作流。

上下文管理
Context Management / 管理AI的上下文信息，避免信息过载

AI上下文过载，记忆丢失。多接口一起写，前面的数据模型、字段规范、返回体、鉴权、校验规则，AI全部记混。一会参数少校验，一会字段名前后不一致，一会返回格式乱套。

类比：就像人同时做多件事，容易顾此失彼。一边打电话一边做饭，可能把盐当成糖，把菜烧糊。
提示：AI的上下文窗口有限，信息太多就会遗忘。单接口开发可以避免这个问题。
批量生成风险
Batch Generation Risks / 批量生成代码的风险和代价

批量一次性生成一堆接口，真实代价很高：bug连环嵌套，排查地狱；代码风格混乱、架构分层崩坏；规范全部失效。生成30分钟，改错+联调+规范整改5～10小时。

类比：就像批量生产的产品，如果模具不对，所有产品都有问题，返工成本巨大。
提示：批量生成适合同质化、模板化的代码，不适合复杂业务逻辑的代码。
混合工作流
Hybrid Workflow / 结合单接口和批量开发的最优工作流

最优工作流（兼顾最快效率+最稳质量）：

  1. 批量生成：架构+目录+公共文件+数据库模型
  2. 批量生成：所有通用CRUD基础接口
  3. 逐个开发：复杂业务个性化接口
  4. 最后统一让AI：格式化代码、补注释、整理文档
类比：就像工厂的生产线，标准件批量生产，定制件单独加工，最后统一组装。
提示：混合工作流的关键是"分类处理"，不同类型的代码用不同的开发策略。
可以批量生成的代码
Batch-Generated Code / 适合批量生成的代码类型

可以批量让AI生成的（放心批量，效率拉满）：

  • 项目整体目录结构
  • 全局公共基础文件（异常捕获、日志、配置、跨域、工具类）
  • 数据库表结构、实体模型、DTO结构体
  • 同质化CRUD接口（列表、详情、新增、编辑、删除）
类比：就像标准件，规格统一，可以批量生产。
提示：这些代码逻辑模板完全一致，可以一次性让AI全套生成，不会乱。
必须单独写的代码
Individually Written Code / 必须单独开发的代码类型

必须"一次一个"单独写的：

  • 业务复杂、有流程、有判断、有状态流转的接口
  • 权限逻辑特殊、数据联动复杂的接口
  • 有事务、异步、缓存、特殊业务规则的接口
类比：就像定制件，每个都有特殊要求，需要单独加工。
提示：这些代码逻辑复杂，AI容易出错，必须逐个开发、逐个测试。

Chapter 08 — 多AI并行开发模式

本章将介绍高阶最优解：多AI并行开发模式，实现高效、稳定、规范、可维护的项目开发。

模块拆分策略
Module Decomposition Strategy / 将项目拆分为独立模块的策略

按"业务模块"拆分AI任务。举例：

  • AI会话A：用户模块（用户表、登录注册个人信息前后端全套）
  • AI会话B：权限角色模块（角色、菜单、权限前后端）
  • AI会话C：文件资源模块（上传、存储、列表前后端）
  • AI会话D：核心业务模块XX
类比：就像建筑工地，不同的施工队负责不同的区域，互不干扰，同时施工。
提示：模块拆分要考虑业务关联性，关联紧密的功能放在同一个模块，避免跨模块调用。
并行开发模式
Parallel Development Mode / 多个AI同时开发不同模块的模式

你同时打开多个对话，分别下发任务，所有AI同时开工，效率直接拉满，比单AI串行快几倍。每个AI=一个独立开发人员，各管各的模块，各写各的表、接口、页面，互不干涉。

类比：就像多人协作，每个人负责自己的部分，同时工作，最后合并成果。
提示：并行开发的前提是"规范统一、接口契约先行"，否则各做各的，最后无法合并。
角色分工
Role Division / 不同AI的角色分工

每个AI的工作范围：只允许做三件事：

  1. 自己模块的数据表CRUD使用（结构不动，只读写）
  2. 自己模块的后端接口业务实现
  3. 自己模块的前端页面/组件/交互

禁止：改全局配置、改其他模块代码、私自加依赖、改表结构。

类比：就像公司部门，市场部只做市场，技术部只做技术，财务部只做财务，互不越界。
提示：角色分工要明确，边界要清晰，否则AI会"越权"修改其他模块的代码。
协作边界
Collaboration Boundary / AI之间的协作边界和规则

协作边界规则：所有人遵守全局隔离规则——只改自己模块，不动全局、不动别人代码、不动已定库表结构。这是保证多AI并行开发不冲突的关键。

类比：就像道路分隔线，每个车道有自己的行驶范围，不能越线行驶，否则会撞车。
提示：协作边界要在开发前定好，并且强制执行。如果AI越界，必须打回重写。
前后端配合模式
Frontend-Backend Collaboration Mode / 前后端AI的协作模式

两种模式任你选：

  • 模式1：单AI全栈负责一个模块（中小型项目推荐）
    一个AI包揽：本模块后端+本模块前端，分工简单，你合并代码即可。
  • 模式2：前后端工种拆分（大型项目推荐）
    一批AI只做所有模块的后端，一批AI只做所有模块的前端，前后端团队并行，按提前约定好的API联调。
类比：模式1就像全能工匠，一个人完成所有工序；模式2就像流水线，每个人只负责一道工序。
提示：选择模式要根据项目规模和团队配置。中小型项目用模式1更简单高效，大型项目用模式2更专业规范。

Chapter 09 — UI设计流程

本章将介绍UI设计的人机协作流程，确保UI风格统一、规范先行。

UI规范先行
UI Standards First / UI设计必须在开发前完成

UI先行，先定视觉规范→先出全部页面UI稿→再动手写前后端业务代码。不按这个顺序，前端必样式混乱、各页面风格割裂、后期大规模返工。

类比：就像装修房子，先定装修风格、选好材料，再开始施工。如果边装修边定风格，最后可能四不像。
提示：UI规范包括：主色调、辅助色、文字颜色、字号层级、圆角大小、间距、阴影、边框风格。
组件库选型
Component Library Selection / 选择UI组件库

人负责选定前端UI组件库：

  • 后台系统：Ant Design / Element Plus / Naive UI
  • C端项目：TailwindCSS + 自研组件 / Shadcn UI
类比：就像选择装修材料，用实木地板还是复合地板，用瓷砖还是木地板，这些选择影响整体风格和成本。
提示：组件库选型要考虑：组件丰富度、文档质量、社区活跃度、团队熟悉度。
UI设计分工
UI Design Division / UI设计的人机分工

你来做（决策权、定标准，不可替代）：

  • 选定前端UI组件库
  • 定全局视觉规范（主色调、辅助色、文字颜色、字号层级、圆角大小、间距、阴影、边框风格）
  • 输出全项目页面清单
  • 定布局方案（侧边栏布局、顶部导航、流式布局等）

AI来做（执行层，全部落地干活）：

  • UI绘图AI：专门出高保真页面效果图（页面长什么样、排版、配色）
  • 前端编码AI：拿着UI效果图，写代码1:1还原页面、写样式、写布局
类比：你就像设计总监，定风格、定标准；AI就像设计师和前端工程师，执行落地。
提示：你只做"定规则、定风格、定范围"，不用自己画图、不用写样式。
UI设计时间点
UI Design Timing / UI设计在整个项目中的时间点

完整标准流程，从上到下：

  1. 你：定需求、业务范围、整体技术栈
  2. 你：定全局UI视觉规范+组件库选型+页面清单
  3. UI设计师AI：产出所有页面的UI设计稿（这一步先做完）
  4. 你：设计数据库、定义全部接口契约
  5. 你：搭建前后端基础骨架、全局配置
  6. 多AI并行开工：各路后端AI开发各自模块接口，各路前端AI对照现成UI稿开发各自模块页面
  7. 前后端联调、测试、部署
类比：就像拍电影，先有剧本，再有分镜头脚本，然后拍摄，最后剪辑。顺序不能乱。
提示：UI定稿→接口定稿之后，所有开发人员（AI）都是"照着图纸施工"，不会自由发挥、不会风格乱飘。

Chapter 10 — 工程化与部署运维

本章将介绍如何让项目从"玩具代码"变成企业级可维护项目。

Docker部署
Docker Deployment / 使用Docker容器化部署应用

AI可以生成Dockerfile、docker-compose.yml部署文件，实现一键部署。Docker可以将应用及其依赖打包成容器，保证在不同环境中运行一致。

类比：就像集装箱，把货物打包成标准尺寸的箱子，无论用船运、车运、火车运，都不需要重新打包。
提示：Docker部署要考虑：镜像大小、启动速度、日志管理、数据持久化。
CI/CD配置
Continuous Integration/Continuous Deployment / 持续集成和持续部署

AI可以生成CI/CD配置（GitHub Actions），实现代码提交后自动测试、自动构建、自动部署。CI/CD是现代软件开发的标准实践。

类比：就像流水线，产品从原材料到成品，经过多道工序，每道工序自动检测质量，不合格的自动剔除。
提示：CI/CD配置要考虑：测试覆盖率、构建速度、回滚机制、通知机制。
部署文档
Deployment Documentation / 详细记录部署步骤和配置的文档

AI可以编写完整README、部署文档、环境搭建文档。部署文档是项目交付的重要组成部分，好的部署文档可以让新人快速上手。

类比：就像产品说明书，详细记录了产品的安装步骤、使用方法、注意事项。
提示：部署文档要包括：环境要求、依赖安装、配置说明、启动步骤、常见问题。
Git规范
Git Standards / Git使用的规范和最佳实践

AI可以生成Git提交规范、.gitignore全套配置。Git规范包括：分支命名规范、提交信息规范、代码审查规范。

类比：就像档案管理，每个文件都有编号、分类、归档规则，方便查找和管理。
提示：Git提交信息要清晰描述本次修改的内容，方便后续查找和回滚。

Chapter 11 — 测试与优化

本章将介绍如何保障项目的稳定性和性能。

单元测试
Unit Testing / 测试单个函数或模块的测试

AI可以自动生成单元测试、接口测试脚本。单元测试是保证代码质量的重要手段，可以及早发现bug。

类比：就像工厂的质量检测，每个零件出厂前都要检测，不合格的不能进入下一道工序。
提示：单元测试要覆盖核心业务逻辑，边界条件，异常情况。AI生成的测试用例可能不全面，需要人工补充。
代码审计
Code Audit / 检查代码质量和安全性的审查

AI可以进行代码审计：找出漏洞、冗余代码、性能问题。代码审计是保证代码质量的重要手段。

类比：就像财务审计，检查账目是否合规，是否有漏洞，是否有浪费。
提示：代码审计要关注：安全漏洞、性能问题、代码规范、架构合理性。
性能优化
Performance Optimization / 提升系统性能的优化措施

AI可以优化慢接口、优化数据库查询、给出并发优化方案。性能优化是提升用户体验的重要手段。

类比：就像汽车保养，定期检查、更换机油、清理积碳，让汽车保持最佳状态。
提示：性能优化要先定位瓶颈，再针对性优化。不要盲目优化，可能适得其反。
日志分析
Log Analysis / 分析系统日志，定位问题

AI可以进行报错日志分析：粘贴报错，让AI定位根因+给出修复代码。日志分析是排查问题的重要手段。

类比：就像医生的诊断报告，记录了病人的症状、检查结果，帮助医生定位病因。
提示：日志要记录关键信息：时间、级别、模块、操作、结果。不要记录敏感信息，如密码、token。

Chapter 12 — 顶层设计与规范体系

本章将深入解释"顶层"和"规范"的本质，帮助你建立全局把控的能力。

顶层设计
Top-Level Design / 项目的整体蓝图和骨架

顶层=盖楼的整体蓝图、骨架、大方向（楼盖多高、几层、户型、地基在哪）。这是项目的骨架大局，决定整个产品的上限，AI没有资格改。

顶层设计包括：

  1. 业务顶层（做什么产品？有哪些功能？哪些页面？）
  2. 产品顶层（用户角色、权限逻辑、整体业务流程）
  3. 技术顶层（用什么框架？前后端栈？数据库？部署方式？）
  4. 数据顶层（全部数据表、字段、表与表之间关联关系）
  5. 接口顶层（全部接口路径、请求方式、入参、出参契约）
  6. UI顶层（整体布局、侧边栏/顶部栏、整体视觉大风格）
  7. 工程顶层（项目文件夹整体目录结构、全局基础配置）
类比：就像建筑总图纸，大楼几层、哪里是客厅、哪里是卧室、地基在哪，这些是顶层设计。
提示：顶层设计告诉所有AI：这个项目整体长什么样、整体逻辑是什么、边界在哪里。
统一规范
Unified Standards / 所有AI必须遵守的硬性规则

规范=施工的统一规矩、标准（砖用多大、门窗尺寸、电线怎么走、墙面颜色）。这是所有AI干活必须遵守的硬性规则，防止各个AI自由发挥、乱成一锅粥。

统一规范包括：

  1. 代码规范（命名、注释、缩进、文件夹使用规则）
  2. 后端规范（统一返回体、状态码、异常处理、事务规则）
  3. 前端规范（组件用法、请求封装、页面结构写法）
  4. UI视觉规范（主色调、字号、圆角、间距、阴影、配色）
  5. 数据库规范（禁止私自加表、改字段、改索引）
  6. 协作规范（每个AI只准动自己模块代码，不许碰别人模块）
  7. 文档规范（注释、README书写格式）
类比：就像施工标准，所有门窗统一尺寸、瓷砖统一规格、油漆统一颜色。
提示：规范让10个AI同时并行开发，写出来的代码、页面、样式风格完全一致，不会东拼西凑。
人机协作结构图
Human-AI Collaboration Structure Diagram / 完整的人机协作结构

完整人机协作结构图：

你（总指挥/架构师）
    ↓
1. 设计【顶层全局框架】
2. 制定【全套统一规范】
    ↓
把顶层蓝图+规范文档同时发给所有AI
    ↓
┌─────┬─────┬─────┬─────┐
│AI UI│AI后端1│AI后端2│AI前端1│  多AI并行干活
│设计师│用户模块│权限模块│页面开发│  每人只做自己的任务
└─────┴─────┴─────┴─────┘
    ↓
所有产出全部符合顶层设计、遵守统一规范
    ↓
你最后合并代码、整体验收、联调上线
类比：你=总设计师/项目经理，多个AI=水电工、泥瓦工、设计师、装修工人，同时开工。
提示：如果没有顶层，AI不知道项目整体方向，乱写乱做；如果没有规范，每个AI各搞一套，代码混乱、UI五花八门、完全没法维护。

Chapter 13 — AI自我迭代与Skills体系

本章将介绍如何让AI长期自我迭代、越来越懂你，建立可复用的AI协作体系。

Skill沉淀
Skill Accumulation / 将规则和经验沉淀为可复用的Skill

想让AI长期自己迭代、越来越懂你、不用每次重复讲规则，必须用Skills固化。临时聊天Prompt永远做不到"自我升级"，只有沉淀成可复用、可版本迭代的Skill库才行。

类比：就像公司的制度文档，把经验教训写成制度，新人入职学习制度就能快速上手，不用每个人重新摸索。
提示：你不用每次给AI发一堆规范，让AI自动加载Skill、自动遵守、主动提优化、迭代更新自己。
自我迭代机制
Self-Iteration Mechanism / AI自我迭代升级的机制

AI自我迭代的闭环：

  1. 你先建好固定Skill技能库（顶层架构、开发规范、UI规范、协作规则等）
  2. 所有AI干活强制先加载全套Skill，按规矩做事
  3. AI在开发中发现：规则有漏洞、流程不顺、写法可优化
  4. AI主动向你提交"Skill迭代修改提案"
  5. 你只做审核同意/驳回，同意就更新进Skill库
  6. 后续所有新AI、老AI，自动使用升级后的新版Skill
类比：就像公司的制度优化，员工发现问题、提出建议、领导审批、制度更新、全员执行。
提示：这就是AI自我迭代：规则沉淀到Skill→AI按规干活→AI发现问题→提案优化→你拍板→全局更新，越用越贴合你的开发习惯，越来越稳。
Skill分类
Skill Classification / Skill的分类和结构

只需要定义这6大类核心Skill：

  1. 全局架构Skill：项目目录、分层结构、技术栈选型、模块拆分原则、工程骨架规则
  2. 后端开发规范Skill：接口写法、统一返回体、状态码、异常处理、数据库使用规则、事务/缓存约束
  3. 前端开发规范Skill：项目目录、组件写法、请求封装、路由规则、页面编码约束、语法规范
  4. UI设计规范Skill：全局配色、字号、圆角、间距、阴影、布局模式、组件库使用标准
  5. 多AI协作分工Skill：模块隔离规则、每个AI只能改自己模块、禁止改动全局/其他模块、任务边界约束
  6. 工程化&交付Skill：Git规范、Docker/部署、README文档、注释规范、项目交付标准
类比：就像公司的部门制度，人事制度、财务制度、技术制度、行政制度，每个制度管一个领域。
提示：就这6个，全覆盖，不多不少，长期够用。
Skill使用方法
Skill Usage / 如何使用Skill

极简操作流程：

  1. 一次性建好6个Skill，定型封存
  2. 以后新开任何AI会话，只需要一句话： 
    加载我全部通用项目Skills，严格遵守所有架构、规范、UI、协作约束，
后续所有开发按Skill执行，不得自创规则。
  3. AI自动读取所有规矩，不用你再重复发任何规范
  4. 日常开发中，AI自觉按Skill落地
  5. 有不合理、可优化的地方，让AI主动输出Skill升级建议
  6. 你简单看一眼，同意就更新Skill，后续全部AI自动沿用新版本
类比：就像新员工入职，学习员工手册，工作时就按照手册执行，发现问题可以提建议优化手册。
提示：关键底线（防止AI乱改顶层）：你锁死顶层架构、数据库全局结构、核心UI大风格这几类Skill，AI只能提建议，不能私自修改，必须你审核。细节编码、样式细则、流程优化，可以放权让AI自己迭代。
两层Skill结构
Two-Layer Skill Structure / 全局Skill和角色专属Skill的结构

两层Skill结构（极简）：

第一层：全局通用Skill（全员必载）

所有AI，不管做UI、做后端、做前端、做架构，必须全部加载。内容就是：

  • 多AI协作规则（不许改别人模块、不许私自改全局表结构）
  • 项目基础目录规范
  • 安全底线、命名大原则
  • 交付、注释、Git通用规矩

第二层：4个角色专属Skill（按需加载，只载自己的）

  • 架构产品专属Skill：只管需求拆解、业务建模、库表设计、接口契约、技术选型
  • UI设计专属Skill：只管视觉风格、配色、布局、页面版式、组件库规则
  • 后端开发专属Skill：只管接口写法、分层、返回体、异常、数据库CRUD、缓存事务
  • 前端开发专属Skill：只管页面组件、路由、请求封装、样式写法、UI组件库使用
类比：全局Skill=公司通用规矩，所有人必须遵守；角色专属Skill=岗位说明书，干什么岗位，只看自己的。
提示：举例：做后端的AI，只加载：全局Skill+后端专属Skill，不需要加载UI、不需要加载架构那套规则，干净不混乱。
完整实战流程
Complete Practical Workflow / 从需求到部署的完整流程

完整实战走一遍：从需求→UI→架构→开发→部署→Bug迭代→新增功能

阶段1：需求梳理

  • 上场AI：架构产品AI
  • 加载Skill：全局Skill+架构产品专属Skill
  • 你做：说清楚要做什么产品、有什么功能、有哪些用户角色
  • AI做：帮你梳理完整需求清单、业务流程、页面清单
  • 你做：审核、拍板定稿

阶段2：UI设计

  • 上场AI：UI设计AI
  • 加载Skill：全局Skill+UI设计专属Skill
  • 你做：定主色调、整体布局风格（侧边栏/顶部导航）
  • AI做：按规范出全套页面UI版式、高保真参考稿
  • 你做：审核UI风格，定稿不许再乱改

阶段3：顶层架构+数据库+接口契约

  • 上场AI：架构产品AI（复用同一个）
  • 加载Skill：全局+架构专属
  • 你做：定技术栈、定项目大框架
  • AI做：设计全量数据表、字段、关联关系、全套REST接口文档
  • 你做：最终拍板锁死库表和接口（后续AI不许私自改动）

阶段4：多AI并行开发（核心）

同时开多个AI会话，分工干活：

  • 后端AI1（用户模块）：加载全局+后端专属Skill，只做用户模块数据表使用、后端接口业务
  • 后端AI2（权限模块）：加载全局+后端专属Skill，只做权限自己模块，不碰别人
  • 前端AI1（用户页面）：加载全局+前端专属Skill，只做按UI稿写自己模块页面、对接约定好的接口
  • 前端AI2（权限页面）：加载全局+前端专属Skill
  • 规则：所有人遵守全局隔离规则——只改自己模块，不动全局、不动别人代码、不动已定库表结构

阶段5：工程化+部署

  • 上场AI：任意一个后端AI就行
  • 加载：全局+后端专属Skill
  • AI做：写Dockerfile、docker-compose、部署文档、环境配置
  • 你做：核对部署脚本，直接上线

阶段6：线上Bug修复

  • 哪个模块出Bug，就找对应模块的原AI
  • 加载：全局+自身角色Skill
  • 直接让它查Bug、改代码、自测
  • 你只审核合并即可

阶段7：后续新增功能迭代

  • 先找架构AI：加载架构Skill，先扩新需求、新增表/新增接口
  • 你审核定稿
  • 分给对应前后端AI，按老规矩并行开发
  • 全程沿用已有Skill，不用重新定规则
类比：就像公司项目，需求分析师做需求，设计师做UI，架构师做架构，开发做开发，运维做部署，各司其职，流程清晰。
提示：不用给每一个AI单独建Skill，按4个角色分就够。架构、UI、后端、前端各加载自己的专属，互不干扰、规范统一、并行不乱。从开发到修Bug到迭代新功能，全程复用这套Skill，AI会越来越适应你的习惯。
AI加载指令
AI Loading Command / AI加载Skill的标准指令

以后新开任何AI会话，直接发这句就行：

请加载我的【全局通用Skill】+ 你当前角色对应的专属Skill，
严格遵守所有规范、协作边界、代码与设计标准，
只在自己职责范围内工作，
不私自修改全局架构、数据表结构和其他模块代码。
类比：就像新员工入职，学习员工手册和岗位说明书，明确自己的职责范围。
提示：这句话可以复制粘贴，每次新开AI会话都用，保证所有AI都遵守同一套规则。

Chapter 15 — 常见翻车案例与避坑指南

本章将总结常见的翻车案例，帮助你避免这些坑。

需求模糊翻车
Vague Requirement Failure / 需求太笼统导致AI乱写一通

翻车表现：

  • 你说"做一个后台"，AI产出一堆你不需要的功能
  • 你说"写一个后端"，AI选择的技术栈你不会用
  • 你说"加个用户管理"，AI理解成简单的增删改查，缺少权限、角色、日志等

避坑方法：

  • 需求拆解到模块→功能→接口→字段最小粒度
  • 使用标准Prompt模板，明确项目信息、强制规范、本次任务、输出要求
  • 先出PRD文档，审核通过后再开发
类比：就像点菜，不能只说"来个菜"，要说清楚"宫保鸡丁，少辣，多放花生"。
提示：需求模糊是最大的翻车原因，宁可多花时间拆解需求，也不要匆忙开发。
规范缺失翻车
Missing Standards Failure / 不提前给规范导致代码风格五花八门

翻车表现：

  • 有的接口返回{code: 0, data: {}}，有的返回{status: "success", result: {}}
  • 有的变量用驼峰命名，有的用下划线命名
  • 有的代码有注释，有的代码没注释
  • 有的接口有异常处理，有的接口裸奔

避坑方法：

  • 开发前先定好全套规范：命名规范、代码规范、接口规范、数据库规范
  • 把规范发给AI，强制执行
  • 使用Skill沉淀规范，不用每次重复发
类比：就像开车没有交通规则，红灯停绿灯行靠自觉，必然乱套。
提示：规范是项目的"宪法"，所有代码都必须遵守，不遵守就打回重写。
批量生成翻车
Batch Generation Failure / 一次性让AI写整个项目导致耦合严重、bug成堆

翻车表现：

  • AI上下文过载，前面的规范后面就忘了
  • 代码风格混乱，有的分层清晰，有的逻辑全堆在控制器
  • bug连环嵌套，一个逻辑错，连带一堆接口报错
  • 无法维护，后期改一个功能，影响一片

避坑方法：

  • 使用混合工作流：批量生成标准代码，逐个开发复杂业务
  • 一次只让AI写一个接口/一个文件
  • 写完一个、测试一个、验收一个，问题当场解决
类比：就像吃饭，一口一口吃，不能一口吃成胖子。
提示：批量生成适合同质化代码，不适合复杂业务逻辑。
技术选型翻车
Technology Selection Failure / 让AI决定技术栈导致技术混乱

翻车表现：

  • AI选择了你不会用的框架，学习成本高
  • AI选择了不成熟的技术，bug多、文档少
  • AI选择了不适合项目规模的技术，过度设计或能力不足

避坑方法：

  • 技术栈选型必须人来决定，AI无权选型
  • 技术选型要考虑：团队能力、项目规模、运维成本、长期维护
  • 选型后告诉AI，AI只负责基于选定技术栈落地实现
类比：就像建房子，建筑师决定用钢筋混凝土结构还是木结构，工人只需要按照图纸施工。
提示：技术选型是战略决策，影响项目的长期发展，不能交给AI。
审核缺失翻车
Missing Review Failure / 不人工审核直接使用导致业务漏洞、安全隐患

翻车表现：

  • 业务逻辑错误，比如库存扣减逻辑不对
  • 权限控制缺失，普通用户能访问管理员接口
  • 安全漏洞，比如SQL注入、XSS攻击
  • 边界异常未处理，比如除零错误、空指针异常

避坑方法：

  • AI所有产出，人工必审核兜底
  • 重点审核：业务逻辑、权限控制、安全漏洞、边界异常
  • 使用单元测试、代码审计工具辅助审核
类比：就像医生开处方，AI可以帮忙写病历、查资料，但最终的处方必须由医生审核签字。
提示：人工审核是质量保障的最后一道防线，不能省略。
