Chapter 01 — AI时代编程新范式

本章将帮助你理解AI如何改变编程方式，建立正确的人机协作思维，明确程序员在AI时代的新角色定位。

AI辅助编程
AI-Assisted Programming / 使用AI工具辅助软件开发过程

AI辅助编程是指利用大语言模型（如GPT-4、Claude、GitHub Copilot等）来辅助程序员完成编码、调试、文档编写等工作的编程方式。这不是让AI替代程序员，而是让AI成为程序员的"超级助手"。

类比：就像建筑师有了智能设计助手，可以快速生成建筑草图、计算结构承重、优化空间布局，但最终的创意、决策和责任仍然在建筑师手中。
💡 提示：AI辅助编程的核心是"人机协作"，不是"AI替代"。程序员需要从"代码编写者"转变为"代码指挥者"。
人机协作
Human-AI Collaboration / 人类与AI协同完成复杂任务的工作模式

人机协作是一种新型工作模式，人类负责顶层决策、架构方向、规范标准、风险边界、业务逻辑，AI负责细节落地、代码编写、文档产出、重复劳动、问题排查。主次一旦颠倒，AI产出必然混乱、不规范、漏洞多、无法维护；主次摆正，就能做到高效、稳定、可迭代、工程化。

类比：就像导演与剧组的关系。导演（人）负责创意、剧本、选角、整体把控，剧组（AI）负责摄影、灯光、道具、后期制作。导演不需要亲自扛摄像机，但必须清楚自己想要什么画面。
💡 提示：人机协作的关键是"人抓大脑，AI做手脚"。人掌握核心控制权，AI负责执行落地。
角色定位
Role Positioning / 在AI时代程序员需要重新定义自己的职责和价值

在AI时代，程序员的角色从"代码编写者"转变为"产品经理+架构师+UI总监+项目总指挥"。程序员只做"定方向、定标准、定契约、拍板确认"，剩下所有落地干活全丢给AI。具体来说，程序员的核心工作包括：

  1. 敲定业务需求：定做什么、不做什么，有哪些页面、哪些功能、角色权限
  2. 敲定整体技术架构：选前后端栈、项目分层、部署方式、整体框架结构
  3. 敲定UI全局风格规范：主色调、组件库、排版风格、圆角/间距/字体规范、整体布局
  4. 敲定数据库+接口契约：表结构、字段、关联关系；每个模块的接口路径、入参出参
  5. 定代码规范+项目目录骨架：全局目录、命名规则、返回体格式、代码分层
类比：就像从"建筑工人"升级为"建筑设计师+项目经理"。以前你需要亲自搬砖、砌墙，现在你只需要画图纸、定标准、监督施工质量。
💡 提示：角色转变的核心是从"执行者"变为"决策者"。你不需要深陷写业务代码、画UI、抠样式、写接口这种琐事，只把控顶层和规范就行。

Chapter 02 — 人机协作五大铁律

本章将介绍人机协作的五条核心铁律，这些铁律适用于所有开发场景，是避免翻车、保证质量的基础保障。

需求拆解原则
Requirement Decomposition Principle / 需求不拆细，绝不让AI动手

模糊指令（做一个后台、写一个后端）= AI产出垃圾；必须拆解到模块→功能→接口→字段最小粒度。这是人机协作的第一条铁律，也是最容易被忽视的原则。

类比：就像装修房子，你不能只说"给我装修一下"，而要说清楚"客厅铺木地板，卧室刷浅蓝色墙漆，厨房装整体橱柜，卫生间做干湿分离"。指令越具体，结果越符合预期。
💡 提示：需求拆解的标准是"AI不需要猜测，只需要执行"。如果AI还需要问你"具体要什么功能"，说明需求还不够细。
规范先行原则
Standards First Principle / 规范先行，开发后置

先把目录、命名、接口、返回体、代码风格、安全规则全部定好，发给AI强制执行，不允许AI自创风格。规范是项目的"宪法"，所有代码都必须遵守。

类比：就像开车前先定交通规则。红灯停、绿灯行、靠右行驶、限速60，这些规则必须先定好，否则路上必然混乱。代码规范就是项目的"交通规则"。
💡 提示：规范一旦定好，就要严格执行。如果AI产出的代码不符合规范，必须打回重写，不能妥协。
任务拆分原则
Task Decomposition Principle / 一次只给AI一个小任务

禁止让AI一次性写完整项目、整套架构；按"模块→接口→函数"分批交付，可控、易纠错。这是保证代码质量的关键原则。

类比：就像吃饭要一口一口吃，不能一口吃成胖子。给AI的任务太大，它就会"消化不良"，产出的代码质量必然下降。
💡 提示：任务拆分的粒度标准是"一次对话能完成，一次测试能验证"。如果一个任务需要多轮对话才能说清楚，说明任务还不够小。
架构控制原则
Architecture Control Principle / 架构&技术栈人全权决定，AI无权选型

用什么框架、数据库、中间件、部署方案人定，AI只负责基于选定技术栈落地实现。这是保证项目可控性的核心原则。

类比：就像建房子，建筑师决定用钢筋混凝土结构还是木结构，工人只需要按照图纸施工。如果让工人自己决定用什么材料，房子可能建歪。
💡 提示：技术选型是战略决策，影响项目的长期发展。AI不了解项目的业务背景、团队技术栈、运维能力，不能让它做这种决策。
人工审核原则
Manual Review Principle / AI所有产出，人工必审核兜底

业务逻辑、权限控制、安全漏洞、边界异常，AI考虑不全，必须人工校验再合并使用。这是质量保障的最后一道防线。

类比：就像医生开处方，AI可以帮忙写病历、查资料，但最终的处方必须由医生审核签字。因为医生了解患者的具体情况，AI不了解。
💡 提示：人工审核的重点是业务逻辑、权限控制、安全漏洞、边界异常。这些是AI最容易出错的地方，也是最容易造成严重后果的地方。

Chapter 03 — 需求梳理阶段

本章将介绍在需求梳理阶段如何让人机协作实现"全面无遗漏"，确保需求分析的质量和完整性。

PRD文档
Product Requirements Document / 产品需求文档，描述产品功能、用户角色、业务流程的文档

PRD文档是需求梳理阶段的核心产出物，包含项目背景、用户角色、核心功能、非功能需求、异常场景、版本迭代规划。AI可以将你的口语需求，转成标准PRD文档、用户故事、功能清单。

类比：就像建筑图纸，PRD文档是项目的"蓝图"，所有人（包括AI）都按照这个蓝图施工。
💡 提示：PRD文档的关键是"结构清晰，不要随意增删业务范围"。AI容易自作主张添加或删除功能，需要人工把控。
用户故事
User Story / 从用户角度描述需求的简短描述

用户故事是敏捷开发中常用的需求描述方式，格式为"作为一个[角色]，我想要[功能]，以便[目的]"。AI可以帮助梳理正向+异常边界场景，AI擅长补人类容易忽略的边缘情况。

类比：就像编剧写剧本，每个场景都要考虑"角色想做什么，为什么这么做，会遇到什么问题"。
💡 提示：AI梳理的用户故事需要人工审核，特别是业务逻辑和边界条件，AI可能理解有偏差。
功能清单
Feature List / 列出所有功能点的清单

功能清单是需求梳理的重要产出，包含所有功能点、优先级、依赖关系。AI可以帮助输出流程图、业务结构图、功能脑图，拆解迭代版本：一期/二期功能拆分。

类比：就像购物清单，列清楚要买什么、先买什么、后买什么，避免遗漏。
💡 提示：功能清单要按照"核心功能→辅助功能→优化功能"的优先级排序，AI可能把所有功能都当成同等重要。
指挥话术模板
Command Template / 标准化的AI指令模板

需求梳理阶段的标准Prompt模板：

基于我提供的业务需求，按照互联网标准PRD格式输出，包含：
项目背景、用户角色、核心功能、非功能需求、异常场景、版本迭代规划，
结构清晰，不要随意增删业务范围。
类比：就像给AI一个"填空题"，让它按照固定格式填写内容，避免AI自由发挥。
💡 提示：使用模板时，要把你的业务需求具体描述清楚，AI才能产出高质量的PRD文档。

Chapter 04 — 架构与技术设计

本章将介绍在架构与技术设计阶段如何实现"结构规范、架构稳定"，确保项目的技术基础扎实可靠。

技术栈选型
Technology Stack Selection / 选择项目使用的技术组合

前后端技术栈（如：React+FastAPI+MySQL+Redis）、项目架构模式（单体/分层架构/微服务）、整体模块拆分（用户、权限、文件、业务、日志等）、部署环境（Linux服务器、Docker、反向代理、域名HTTPS）、安全方案（鉴权JWT、接口防刷、跨域、数据脱敏）等，这些都必须由人来决定。

类比：就像选择建筑材料，用钢筋混凝土还是木结构，用砖墙还是玻璃幕墙，这些决策影响建筑的承重、成本、工期、维护难度。
💡 提示：技术选型要考虑团队能力、项目规模、运维成本、长期维护等因素，AI不了解这些背景，不能让它做决策。
模块拆分
Module Decomposition / 将系统拆分为独立的业务模块

AI可以输出整体项目目录结构（强制统一，后续所有代码都按此目录）、数据库设计（ER图、数据表、字段设计、索引、关联关系）、全局基础架构封装（全局异常处理、日志封装、配置中心、跨域配置）、绘制架构拓扑图、数据流图、权限结构图。

类比：就像把大房子分成客厅、卧室、厨房、卫生间，每个空间有独立的功能，但又相互连接。
💡 提示：目录结构一旦确定，全程不许AI私自新增/修改文件夹。这是保证项目结构统一的关键。
数据库设计
Database Design / 设计数据表结构、字段、索引、关联关系

数据库设计是架构阶段的核心工作，包括ER图、数据表、字段设计、索引、关联关系。AI可以帮助设计数据库，但最终的表结构必须由人来审核定稿。

类比：就像设计建筑的承重结构，哪里是柱子、哪里是梁、哪里是墙，这些决定了建筑的稳定性和扩展性。
💡 提示：数据库设计要考虑业务增长、查询性能、数据一致性等因素。AI设计的表结构可能存在范式问题、索引缺失、关联关系不合理等问题，需要人工审核。
全局架构封装
Global Architecture Encapsulation / 封装全局通用的基础功能

全局架构封装包括全局异常处理、日志封装、配置中心、跨域配置等。这些是项目的基础设施，所有模块都会使用。

类比：就像建筑的供水供电系统，每个房间都需要用水用电，所以需要统一设计和施工。
💡 提示：全局架构封装要考虑扩展性、性能、安全性。AI可能只考虑当前需求，忽略未来的扩展。

附录 — 人机协作完整流程清单

完整流程图

1. 需求梳理阶段
   你：定业务目标、用户角色、核心功能优先级、盈利/使用场景、项目范围边界
   AI：转成标准PRD文档、用户故事、功能清单；梳理正向+异常边界场景；
       输出流程图、业务结构图、功能脑图；拆解迭代版本

2. 架构与技术设计阶段
   你：定前后端技术栈、项目架构模式、整体模块拆分、部署环境、安全方案
   AI：输出整体项目目录结构、数据库设计、全局基础架构封装；
       绘制架构拓扑图、数据流图、权限结构图

3. 接口设计阶段
   你：定接口风格、全局统一返回体格式、全局状态码定义、鉴权方式、请求头公共参数、分页通用规则
   AI：按模块批量生成OpenAPI/Swagger接口文档；
       逐个接口定义：路径、请求方式、入参、出参、校验规则；
       生成前后端对接接口清单

4. UI设计阶段
   你：定UI组件库、全局视觉规范、页面清单、布局方案
   AI：产出所有页面的UI设计稿

5. 编码开发阶段
   你：定强制约束文件、拆分任务
   AI：按规范编写代码

6. 工程化与部署阶段
   AI：生成Dockerfile、docker-compose.yml、CI/CD配置；
       编写完整README、部署文档、环境搭建文档；
       配置日志、进程守护、Git提交规范、.gitignore

7. 测试与优化阶段
   你：制定核心业务测试用例
   AI：自动生成单元测试、接口测试脚本；
       代码审计：找出漏洞、冗余代码、性能问题；
       优化慢接口、优化数据库查询、给出并发优化方案；
       报错日志分析：定位根因+给出修复代码

检查清单

需求梳理阶段检查清单：

  • PRD文档完整（项目背景、用户角色、核心功能、非功能需求、异常场景、版本迭代规划）
  • 用户故事清晰（角色、功能、目的）
  • 功能清单完整（功能点、优先级、依赖关系）
  • 流程图、业务结构图、功能脑图齐全
  • 迭代版本拆分合理

架构与技术设计阶段检查清单：

  • 技术栈选型合理（前后端技术栈、数据库、中间件、部署环境）
  • 项目架构模式明确（单体/分层架构/微服务）
  • 模块拆分清晰（用户、权限、文件、业务、日志等）
  • 数据库设计完整（ER图、数据表、字段设计、索引、关联关系）
  • 全局基础架构封装完整（全局异常处理、日志封装、配置中心、跨域配置）

接口设计阶段检查清单：

  • 接口风格统一（RESTful规范）
  • 全局统一返回体格式定义
  • 全局状态码定义完整
  • 鉴权方式明确
  • OpenAPI/Swagger接口文档完整
  • 前后端对接接口清单齐全

编码开发阶段检查清单：

  • 强制约束文件完整（项目完整目录结构、命名规范、代码注释规范、分层规范、代码格式规范）
  • 任务拆分合理（全局公共类→基础工具类→数据模型→单个接口业务逻辑→前端页面/组件）
  • 代码符合规范（命名、注释、格式、分层）
  • 代码带完整注释
  • 代码可直接运行、依赖明确
  • 逻辑分层清晰、解耦
  • 不写冗余代码、不私自引入新依赖
